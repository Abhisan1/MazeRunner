## Development Usage

### What Tasks AI Assisted With

- **Maze layout design** — Designing the grid structure, verifying paths were connected and walkable, and catching a critical error where the checkpoint was placed inside a wall (unreachable by the player)
- **Code generation** — All C# scripts were generated with AI assistance as a starting point
- **Architecture decisions** — AI suggested building UI programmatically instead of manually in the Unity Inspector, and using a grid array for maze generation instead of hand-written coordinates
- **Bug diagnosis** — When errors appeared in the Unity Console, AI identified the cause and suggested targeted fixes
- **Unity 6 compatibility** — AI helped navigate breaking changes in Unity 6, particularly the new Input System package replacing legacy Input.GetAxis()
- **Mobile input strategy** — AI and developer jointly decided to switch from accelerometer to D-pad after testing showed accelerometer required extreme phone tilt for usable movement

### What Code Was Generated by AI

All scripts were AI-generated as a starting point and then tested and validated:
- `MazeGenerator.cs` — Editor script that builds maze from a 27×27 grid array
- `PlayerController.cs` — Physics movement with keyboard (PC) and D-pad (Android) input
- `CameraController.cs` — Dual camera modes with smooth follow and fixed top-down
- `GameManager.cs` — Game state, scoring, checkpoint, leaderboard logic
- `UIManager.cs` — Programmatic UI builder for all 5 screens
- `TriggerHandler.cs` — Zone and obstacle collision detection
- `AIObstacle.cs` — Patrol-based AI movement between two points
- `ObstacleSpinner.cs` — Rotation animation for static obstacles
- `MobileJoystick.cs` — Singleton input state manager for D-pad
- `DpadButton.cs` — Individual button press/release handler
- `DpadSetup.cs` — Programmatic D-pad canvas builder

### What Was Done Manually (Developer Contribution)

- Setting up the Unity project, configuring Unity Hub, installing Android Build Support
- Creating folder structure (Scripts, Materials, Prefabs, Scenes)
- Attaching scripts to GameObjects in the Unity Editor
- Configuring Rigidbody properties — mass, drag, angular drag, freeze rotation constraints
- Setting up colliders and IsTrigger flags on zones and obstacles
- Tagging GameObjects (Player tag, Obstacle tag)
- Connecting the Player reference to CameraController via Inspector drag-and-drop
- Setting canvas scaler reference resolution and match mode
- Testing every system after each change and reporting bugs back for fixes
- Making all gameplay design decisions — ball speed, AI patrol range, penalty amount, scoring rate
- Deciding to switch from accelerometer to D-pad based on real device testing
- Catching the checkpoint-in-wall placement error during design review
- Managing the build process — switching platform, configuring package name, building APK
- Installing and testing APK on a real Android device multiple times

## Prompt Examples

**Prompt 1 — Maze Generator:**
"Write a Unity Editor script with a MenuItem 'MazeRunner/Build Maze' that reads a 27x27 integer grid where 0 is wall and 1 is open corridor, and places Unity cube GameObjects for every wall cell. Position each cube as (col minus 13, 0.5, row minus 13) so the maze is centered at world origin. Also place a green StartZone, red FinishZone, and yellow Checkpoint as flat trigger cubes at specific grid coordinates. Group everything under a parent GameObject called Maze."

**Prompt 2 — D-Pad Input System:**
"Write three Unity scripts: MobileJoystick (singleton that stores a Vector2 input value), DpadButton (implements IPointerDownHandler and IPointerUpHandler to call MobileJoystick.SetInput with a direction vector on press and Vector2.zero on release), and DpadSetup (MonoBehaviour that programmatically creates a Canvas with four directional buttons arranged in a cross pattern at the bottom-left of the screen, each with a DpadButton component and correct direction assigned). Only build the canvas on mobile platform."

**Prompt 3 — Programmatic UI Builder:**
"Rewrite UIManager to build all UI panels in code using Awake(). Create helper methods MakePanel, AnchoredText, AnchoredButton, and AnchoredSlider. Use normalized anchor positions (Vector2 between 0 and 1) so UI elements scale correctly on all screen sizes. Build StartPanel with title, controls instructions, sensitivity slider, and Play button. Build GamePanel with live score top-left, best score below it, CAM and PAUSE buttons top-right, and a centered notification text for checkpoint/penalty messages. Build PausePanel, WinPanel (with top 5 leaderboard), and LosePanel. All panels toggle visibility via SetActive."

## Validation Process

### How AI Generated Code Was Verified

- **Read before pasting** — Every script was read through before being pasted into Visual Studio to understand what it was doing and why
- **Console monitoring** — After every script change, the Unity Console was checked for errors and warnings before proceeding to the next step
- **Incremental testing** — Each system was tested in isolation before building on top of it. Movement first, then camera, then obstacles, then scoring, then UI
- **Real device testing** — APK was installed and tested on a physical Android device multiple times, not just the Unity Editor
- **Gameplay verification** — Every requirement from the PDF was manually tested and confirmed working before submission

### Incorrect Outputs Received

1. **Input System error** — AI initially used legacy Input.GetAxis() which threw InvalidOperationException in Unity 6. Fixed by switching to Keyboard.current from UnityEngine.InputSystem
2. **Maze coordinate mismatch** — First maze generator used hand-written wall coordinates that did not match the visual layout. Identified visually in Scene view and fixed by switching to grid-array approach
3. **Missing StartGame method** — UIManager referenced GameManager.StartGame() which did not exist initially. Caught by compiler error, fixed by adding the missing method
4. **Checkpoint placed inside wall** — Initial checkpoint position was inside a walled-off area of the maze. Caught during design review before building, corrected to center of open corridor
5. **DontDestroyOnLoad duplicate objects** — On scene restart, persistent objects caused duplicate balls, canvases, and EventSystems. Identified during Android testing, fixed by removing DontDestroyOnLoad and adding singleton duplicate-destruction
6. **Up/Down button direction reversed** — D-pad Up button moved ball backward. Identified during Android testing, fixed by changing move.z = -joy.y to move.z = joy.y in PlayerController
7. **Accelerometer UX failure** — Accelerometer input worked technically but required extreme phone tilt for movement, making the game unplayable. Identified through real device testing and resolved by switching to D-pad buttons

### How Issues Were Identified and Fixed

- Compiler errors were read carefully from the Unity Console, identifying exact file, line number, and nature of the problem
- Each error was described back to the AI with the full error message for a targeted fix
- After fixes, the game was played again to verify the fix worked without introducing new problems
- Design issues were caught through visual inspection of the Scene view
- Android-specific issues were caught exclusively through real device testing — the Unity Editor alone was insufficient for mobile validation
